# 需求清单

这份文档只写“用户真正想要什么”，不写具体代码怎么实现。当前可用功能和限制以仓库 README 为准。

## 核心目标

做一个本地优先的 Windows 语音输入工具：

- 尽量离线运行。
- 低延迟，适合日常输入。
- 当前优先适配 7840HS 迷你主机；4090 台式机作为后续服务端增强路线。
- 可以按住热键说话，松开后自动把文字放到当前光标位置。
- 不破坏用户原来的剪贴板内容。
- 能处理普通音频文件转文字。
- 能快速记录灵感/待办/素材，并按关键词自动保存到指定文件夹。
- 能配置热词和模型，适应用户自己的输入习惯。
- 能按使用场景路由模型：日常语音输入优先快，文件转写/长音频可以后台慢慢跑。
- 后续要能接 API 或自定义服务，例如兼容类似“硅基流动给沉浸式翻译接入”的自定义接口思路。

## 已确认需求

### 语音输入

- 按住一个热键开始录音。
- 松开热键停止录音并识别。
- 识别结果自动粘贴到当前光标位置。
- 自动粘贴时必须保护原剪贴板。
- 可以选择只复制到剪贴板，不自动粘贴。
- 识别结果可以保存，录音 wav 默认不保留；需要排错、留证据或用户主动开启时，才保留每次录音文件。

### 快速记录

- 说出一段内容后，程序可以把识别文本保存为记录文件。
- 可以通过关键词决定保存位置。
- 关键词应该主要在识别文本开头附近生效，不应该在整段文本任意位置都触发，避免用户后面随口提到关键词导致误保存。
- 关键词匹配范围应可配置，例如只检查开头若干个字符。
- 不同关键词可以保存到不同文件夹。
- 多个关键词也可以指向同一个文件夹。
- 每条关键词规则应可配置是否把命中的关键词从最终保存文本里去掉。
- 如果没有命中关键词，保存到默认收件箱文件夹。
- 记录文件名应自动生成，包含时间，避免覆盖。
- 第一版可以用命令行配置关键词规则；后续再做 GUI。
- GUI 里应能看到快速记录状态，并能启用或关闭快速记录；从 GUI 启动热键听写时，这个开关应真正影响后台 `hold-to-talk` 是否带 `--quick-note`。

示例：

```text
关键词“灵感” -> notes\ideas\
关键词“待办” -> notes\todo\
关键词“小说” -> notes\novel\
关键词“素材”“摘录” -> notes\materials\
未命中 -> notes\inbox\
```

### 关键词与热词

- 关键词用于“路由”：决定识别结果保存到哪个文件夹。
- 热词用于“识别”：提高特定词汇、人名、项目名、术语的识别稳定性。
- 两者不是一回事，但都需要可配置。
- 第一版热词可以先作为配置项保存，并传给支持热词的后端；当前后端不支持时，要在文档/诊断里说明。
- 热词后续应支持权重：常用、刚确认过的热词权重提高，长期未使用或经常被判定为同音误触的热词权重降低。
- 热词权重第一版应优先使用可解释的数学规则，不依赖 AI；AI 或本地文本后处理只能作为可选的误触/确认信号来源。
- 设计草案记录在 `docs/hotword-weighting-design.md`。

### 剪贴板

- 自动粘贴时临时占用剪贴板。
- 粘贴完成后恢复原剪贴板。
- Windows 上应尽量恢复多种剪贴板格式，不只是文本。
- 如果某些特殊格式不能恢复，要在日志里能看出来。

### 文件转录

- 输入 wav 等音频文件，输出文字。
- 支持拖动/拖拽文件触发识别。
- 支持保存 `.txt`。
- 支持基础 `.srt` 字幕。
- 后续如果模型能提供时间戳，就用真实时间戳；没有时间戳时，至少输出一个覆盖整段音频的基础字幕块。

### 模型配置

- 用户可以查看当前模型。
- 用户可以手动指定模型。
- 用户可以切回自动选择。
- 默认配置应支持自动分配：CPU 机器优先选择轻量本地模型；有 NVIDIA 显卡时，可根据显存自动选择更合适的模型。
- 不同任务可以有不同模型路由策略：语音输入/听写优先低延迟，文件转写和长音频默认可以后台处理并偏向准确率。
- 如果模型资源不足，要给警告，而不是静默失败。
- 模型配置应该写入配置文件，避免每次启动都重新传参数。
- 模型应该先进入候选池，经过本机测速后再下载和推荐；不要看到新模型就立刻下载。
- 推荐列表应以 7840HS 的真实推理速度、准确率、稳定性和安装复杂度为依据。
- 4090 机器应作为后续服务端/远程推理路线，而不是日常客户端的默认假设。
- Qwen3-ASR-GGUF 应作为下一步重点候选，先测 0.6B，再考虑 1.7B。
- Fun-ASR 1.5 应作为方言特殊项目跟踪：优先关注普通话、粤语、吴语、闽语、客家话、湘语、赣语等中文方言/口音场景。
- NVIDIA Nemotron Speech Streaming 0.6B 可以作为英文流式实验模型跟踪，但不应替换中文默认模型。

### 客户端与服务端

- 客户端优先跑在 7840HS 迷你主机上，负责热键、录音、粘贴、剪贴板保护、快速记录和配置。
- 服务端后续可以跑在 4090 机器上，负责高准确率模型、长音频、批处理和远程 ASR。
- 第一版先不急着做网络服务，但架构不能把“所有模型一定在本机跑”写死。

### API 与自定义服务

- 本地识别是默认优先路径，但架构要预留 API 接入。
- API 接入应尽量做成可配置 provider，而不是把某一家服务写死。
- 配置里至少要能表达 provider 名称、base URL、模型名、API key 环境变量名和超时时间。
- 后续接入硅基流动、OpenAI 兼容接口或其他自定义服务时，应走同一套 provider 配置。
- 涉及 API key 时，不应该把明文密钥写进项目代码或普通配置文件，优先使用环境变量。
- API 后处理不只是“润色”，还应该能做格式排版、标点补全、口语清理、同音字修正、ASR 误识别修正和专有名词修正。
- API 后处理应支持增强上下文，但必须可控、可关闭，不能默认把大量历史内容都发给外部服务。
- 增强上下文第一版默认只使用最近 3 条听写文本，并且要有总字符数上限。
- 增强上下文应支持术语表，优先用于人名、项目名、模型名、设备名、软件名等稳定词汇。
- 热词可以先作为术语表来源复用，但文档里要说明：热词偏向识别阶段，术语表偏向 API 后处理阶段。
- 后续可以增加智能压缩，把更长历史压成短摘要；但这个能力默认关闭，不能影响短句热键输入速度。
- 外部 API 后处理默认只发送文字，不发送音频。
- GUI 里应能看到 API 整理、增强上下文、术语表和失败退回原文这些状态，避免用户不知道输出为什么被改写。

### 学习和维护

- 每个重要功能都要有新手能看懂的文档。
- 维护任务应该有可追踪的计划、状态和验证记录。
- 每次改动后要运行对应测试和环境体检命令。

## 暂缓需求

- 长音频说话人分离。
- 更大模型和 GPU 加速。

这些不是不要，而是等命令行闭环稳定后再做。
