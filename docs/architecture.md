# 架构骨架

## 当前分层

- `model_selector.py`：纯策略层。只根据任务、语言、硬件和用户偏好选择模型。
- `config.py`：配置层。负责读写 JSON 配置，默认使用标准库，先不引入额外依赖。
- `audio_capture.py`：录音层。负责枚举麦克风、录音并保存 wav，不直接调用 ASR 模型。
- `asr.py`：ASR 合约层。定义后端需要实现的输入、输出和异常。
- `text_output.py`：安全输出层。负责保存文本和复制剪贴板，先不直接向当前窗口键入。
- `subtitles.py`：字幕输出层。负责把识别结果格式化为基础 SRT。
- `backends.py`：后端注册表。真实后端接入前，默认注册为“不可用”，避免 UI 层遇到模糊导入错误。
- `sherpa_backend.py`：第一个真实后端适配器。负责把项目内部的转录请求转换成 `sherpa-onnx` 调用。
- `app.py`：应用服务层。把配置、硬件检测、模型选择和后端调用串起来。
- `cli.py`：早期验证入口。可先用命令行检查硬件检测、模型列表和推荐结果。

## 客户端/服务端方向

当前先做单机命令行闭环，但架构要允许后续拆成客户端和服务端。

客户端运行在 7840HS 迷你主机上，负责：

- 热键监听。
- 录音。
- 剪贴板保护和自动粘贴。
- 快速记录关键词路由。
- 用户配置和状态展示。

服务端可以运行在 4090 台式机上，负责：

- 重模型 ASR。
- 长音频和批处理。
- 后续局域网 API。

因此 `asr.py` 的后端合约不要绑定具体运行位置。未来本地后端、Foundry Local 后端、API 后端、4090 服务端后端，都应该实现同一类“输入音频，输出转录结果”的接口。

对 `Qwen3-ASR-GGUF` 这类第三方实验路线，当前更推荐的接入方式是：

- 先作为独立实验工具存在；
- 如果后续要接入主项目，优先做“subprocess 适配层”；
- 不要一开始就把第三方内部推理代码直接揉进主项目运行时。

这样做的原因很朴素：

- 第三方工具的依赖栈和当前 `sherpa-onnx` 不一样；
- Windows 打包版可能带来控制台编码、DLL、ffmpeg、GPU 开关等额外问题；
- 用 subprocess 包一层，更容易把风险关在边界外，也更方便将来替换或删除。

## 后续接入顺序

1. 下载 SenseVoice ONNX 模型文件，让 `sherpa_backend.py` 先跑通 wav 文件转录。
2. 用 `listen-once` 验证“录音 -> wav -> 转录”的最小闭环。
3. 增加按住说话热键，默认只复制剪贴板，用户确认后再做自动粘贴。
4. 再做拖拽文件转录、字幕导出和长音频模型。
5. 在 7840HS 上先做模型测速，再决定是否下载 Qwen3-ASR-GGUF 0.6B。
6. 客户端稳定后，再评估 4090 服务端推理。

## 约束

- 模型选择层不导入 UI、录音库或具体推理库。
- 后端失败必须抛出清晰异常，不在 UI 或 CLI 中静默吞掉。
- 第一版不把模型权重打包进程序，只记录来源、许可和下载/安装状态。
- 新模型先登记为候选并测速，不能直接下载一堆权重。
- 用户常用设置通过 `open-voice-input config set ...` 写入 JSON，避免每次命令都重复传设备编号和语言。
